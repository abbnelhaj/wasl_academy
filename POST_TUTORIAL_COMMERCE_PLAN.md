# Post-Tutorial Commerce Plan

## الهدف

بعد الانتهاء من التتوريال، سنحول Wasl Academy من منصة كورسات فقط إلى منصة كورسات بتجربة شراء كاملة:

- سلة مشتريات.
- كورسات مجانية ومدفوعة.
- دفع إلكتروني عبر T-Link: https://tdsp.ly/
- دفع عند الاستلام عبر مندوب، مع ربط درب سبيل لحساب تكلفة التوصيل/التحصيل.
- محفظة داخلية للمستخدم.
- لوحة أدمن لإدارة الطلبات والمدفوعات والوصول للكورسات.

هذه الخطة مؤجلة حتى نكمل التتوريال. لا ننفذها الآن.

## القرار الأساسي

لن نستخدم منطق الاشتراكات مثل التتوريال.

المنطق الصحيح لمنصتنا:

- المستخدم يشتري كورس أو أكثر.
- الكورس قد يكون مجاني أو مدفوع.
- الدفع يكون مرة واحدة لكل طلب.
- الوصول للكورس يعطى بعد تأكيد الدفع أو بعد اعتماد الأدمن لحالات معينة.
- `completedBy` يبقى فقط لمعرفة من أنهى الكورس، وليس لمعرفة من اشترى الكورس.

## حدود المسؤوليات

### Clerk

- تسجيل الدخول.
- معرفة هوية المستخدم `userId`.
- حماية الصفحات والمسارات.

لا نستخدم Clerk لتخزين المشتريات أو المحفظة.

### Sanity

- محتوى الكورسات.
- الموديولات والدروس.
- التصنيفات.
- معلومات العرض مثل العنوان والوصف والصورة.
- حقول التسعير الأساسية في الكورس مثل `accessType`, `price`, `currency`.

لا نخزن الطلبات أو المدفوعات أو رصيد المحفظة في Sanity.

### قاعدة البيانات

نستخدم قاعدة بيانات للبيانات التشغيلية:

- السلة.
- الطلبات.
- عناصر الطلب.
- محاولات الدفع.
- صلاحية الوصول للكورسات.
- المحفظة.
- الشحن/التحصيل.
- سجلات الأدمن.

الخيار المفضل لاحقا: Postgres مع Prisma أو Drizzle.

### T-Link

- معالجة الدفع الإلكتروني.
- إنشاء جلسة/طلب دفع.
- استقبال إشعارات الدفع أو callback/webhook.
- تأكيد نجاح الدفع قبل فتح الكورس.

سنحتاج وثائق API الرسمية قبل التنفيذ.

### درب سبيل

يستخدم لحالات الدفع عند الاستلام أو التحصيل:

- حساب تكلفة التوصيل/التحصيل.
- إنشاء طلب مندوب.
- متابعة حالة المندوب.
- تأكيد أن القيمة تم استلامها.

سنحتاج وثائق API الرسمية قبل التنفيذ.

## الموديلات المقترحة

### Cart

يمثل سلة المستخدم الحالية.

حقول متوقعة:

- `id`
- `userId`
- `status`: `active`, `converted`, `abandoned`
- `createdAt`
- `updatedAt`

### CartItem

عنصر داخل السلة.

حقول متوقعة:

- `id`
- `cartId`
- `itemType`: `course`, `product`, `bundle`
- `itemId`
- `titleSnapshot`
- `priceSnapshot`
- `currencySnapshot`
- `quantity`

ملاحظة: نأخذ snapshot للسعر والعنوان وقت الإضافة/الشراء حتى لا تتغير الطلبات القديمة إذا تغير سعر الكورس لاحقا.

### Order

الطلب النهائي بعد checkout.

حقول متوقعة:

- `id`
- `userId`
- `status`
- `paymentMethod`: `tlink`, `wallet`, `cash_on_delivery`, `manual`
- `subtotal`
- `deliveryFee`
- `discountTotal`
- `total`
- `currency`
- `createdAt`
- `updatedAt`

حالات الطلب المقترحة:

- `pending_payment`
- `paid`
- `pending_cod_collection`
- `cod_out_for_collection`
- `cod_collected`
- `fulfilled`
- `cancelled`
- `failed`
- `refunded`

### OrderItem

عنصر داخل الطلب.

حقول متوقعة:

- `id`
- `orderId`
- `itemType`: `course`, `product`, `bundle`
- `itemId`
- `titleSnapshot`
- `priceSnapshot`
- `quantity`
- `accessGrantedAt`

### PaymentAttempt

كل محاولة دفع.

حقول متوقعة:

- `id`
- `orderId`
- `provider`: `tlink`, `wallet`, `cash_on_delivery`, `manual`
- `providerReference`
- `status`: `pending`, `paid`, `failed`, `cancelled`, `refunded`
- `amount`
- `currency`
- `rawPayload`
- `createdAt`

### Enrollment

صلاحية المستخدم للوصول إلى كورس.

حقول متوقعة:

- `id`
- `userId`
- `courseId`
- `source`: `free`, `order`, `manual_admin`, `promo`
- `orderId`
- `status`: `active`, `revoked`, `refunded`
- `grantedAt`
- `revokedAt`

### WalletTransaction

المحفظة تكون ledger، وليس رقم balance فقط.

حقول متوقعة:

- `id`
- `userId`
- `type`: `credit`, `debit`
- `source`: `topup`, `purchase`, `refund`, `admin_adjustment`, `promo`
- `amount`
- `currency`
- `orderId`
- `adminId`
- `note`
- `createdAt`

الرصيد يحسب من مجموع الحركات.

### ShipmentOrCollection

خاص بدرب سبيل لحالات التوصيل أو التحصيل.

حقول متوقعة:

- `id`
- `orderId`
- `provider`: `darb_sabil`
- `providerReference`
- `status`
- `deliveryFee`
- `customerName`
- `customerPhone`
- `city`
- `address`
- `notes`
- `createdAt`
- `updatedAt`

## تدفق الشراء

### كورس مجاني

1. المستخدم يسجل دخول.
2. يضغط enroll.
3. السيرفر يتأكد أن الكورس `accessType = free`.
4. ينشئ `Enrollment` مباشرة.
5. المستخدم يدخل محتوى الكورس.

### كورس مدفوع عبر T-Link

1. المستخدم يضيف الكورس للسلة.
2. المستخدم يذهب إلى checkout.
3. السيرفر يحسب السعر من Sanity/قاعدة البيانات، وليس من الواجهة.
4. ينشئ `Order` بحالة `pending_payment`.
5. ينشئ `PaymentAttempt` بحالة `pending`.
6. يطلب من T-Link إنشاء عملية دفع.
7. المستخدم يدفع.
8. T-Link يرسل callback/webhook.
9. السيرفر يتحقق من التوقيع/المرجع.
10. السيرفر يحدث `PaymentAttempt` إلى `paid`.
11. السيرفر يحدث `Order` إلى `paid`.
12. السيرفر ينشئ `Enrollment`.
13. المستخدم يحصل على الوصول للكورس.

### الدفع بالمحفظة

1. المستخدم يختار الدفع بالمحفظة.
2. السيرفر يحسب الرصيد من `WalletTransaction`.
3. إذا الرصيد يكفي، ينشئ حركة `debit`.
4. يحدث الطلب إلى `paid`.
5. ينشئ `Enrollment`.

### الدفع عند الاستلام عبر درب سبيل

هذا الخيار مناسب إذا كان الطلب يحتوي شيئا يحتاج توصيل أو تحصيل نقدي.

1. المستخدم يختار الدفع عند الاستلام.
2. النظام يطلب عنوان ورقم هاتف.
3. النظام يستدعي درب سبيل لحساب تكلفة التوصيل/التحصيل.
4. ينشئ `Order` بحالة `pending_cod_collection`.
5. ينشئ طلب تحصيل لدى درب سبيل.
6. المندوب يذهب للزبون ويستلم قيمة الطلب.
7. درب سبيل يؤكد التحصيل أو الأدمن يؤكد يدويا.
8. يتحول الطلب إلى `cod_collected` ثم `paid`.
9. النظام ينشئ `Enrollment`.

قرار مهم لاحقا: هل نفتح الكورس مباشرة بعد إنشاء طلب COD، أو بعد تأكيد التحصيل؟

المقترح الآمن: لا نفتح الكورس إلا بعد تأكيد التحصيل.

## لوحة الأدمن المطلوبة

### إدارة الطلبات

- عرض كل الطلبات.
- فلترة حسب الحالة وطريقة الدفع.
- عرض تفاصيل الطلب.
- تأكيد دفع يدوي.
- إلغاء طلب.
- إعادة محاولة الدفع.
- تحويل طلب إلى مدفوع عند الحاجة.

### إدارة الوصول للكورسات

- إعطاء وصول يدوي لمستخدم.
- إلغاء الوصول.
- معرفة الكورسات التي يملكها المستخدم.
- معرفة المستخدمين الذين يملكون كورس معين.

### إدارة المحفظة

- عرض رصيد المستخدم.
- عرض كل حركات المحفظة.
- إضافة رصيد يدوي مع سبب.
- خصم رصيد يدوي مع سبب.
- منع أي تعديل بدون `AdminAuditLog`.

### إدارة الدفع عند الاستلام

- عرض طلبات COD.
- عرض حالة درب سبيل.
- إعادة إرسال طلب التحصيل.
- تأكيد التحصيل يدويا عند الحاجة.
- تسجيل سبب أي تعديل يدوي.

### التقارير

- إجمالي المبيعات.
- مبيعات حسب طريقة الدفع.
- مبيعات حسب الكورس.
- الطلبات المعلقة.
- طلبات COD غير المحصلة.
- رصيد المحافظ.
- الاسترجاعات.

### سجل الأدمن

أي عملية حساسة تسجل:

- الأدمن الذي نفذ العملية.
- العملية.
- قبل/بعد إن وجد.
- السبب.
- الوقت.

## قواعد الأمان

- لا نثق بالسعر القادم من الواجهة.
- لا نفتح كورس مدفوع إلا من السيرفر.
- لا نعتبر الدفع ناجحا من صفحة success فقط.
- نعتمد على webhook/callback موثوق.
- نستخدم idempotency حتى لا يتكرر فتح الكورس أو خصم المحفظة.
- نحفظ نسخة من payload القادمة من بوابة الدفع للتدقيق.
- نستخدم audit log لأي تعديل يدوي من الأدمن.

## مراحل التنفيذ بعد التتوريال

### المرحلة 1: الأساس التجاري

- اختيار قاعدة البيانات.
- إضافة موديلات `Order`, `OrderItem`, `Enrollment`.
- بناء منطق الوصول للكورسات.
- بناء checkout داخلي بدون بوابات دفع.
- دعم enroll للكورسات المجانية.

### المرحلة 2: السلة

- إضافة Cart و CartItem.
- واجهة السلة.
- صفحة checkout.
- منع تكرار شراء نفس الكورس.

### المرحلة 3: لوحة الأدمن الأولى

- عرض الطلبات.
- تأكيد دفع يدوي.
- إعطاء وصول يدوي.
- عرض كورسات المستخدم.

### المرحلة 4: T-Link

- قراءة وثائق T-Link.
- إنشاء payment session.
- إضافة callback/webhook.
- اختبار حالات النجاح والفشل والإلغاء.
- ربط الدفع بإنشاء Enrollment.

### المرحلة 5: المحفظة

- إضافة WalletTransaction.
- عرض الرصيد للمستخدم.
- الدفع بالمحفظة.
- استرجاع المبالغ للمحفظة.
- أدوات الأدمن لتعديل المحفظة.

### المرحلة 6: الدفع عند الاستلام ودرب سبيل

- قراءة وثائق درب سبيل.
- حساب تكلفة التحصيل/التوصيل.
- إنشاء طلب مندوب.
- متابعة حالة الطلب.
- فتح الكورس بعد تأكيد التحصيل.

### المرحلة 7: التقارير والتحسينات

- تقارير مبيعات.
- تقارير COD.
- كوبونات.
- خصومات.
- فواتير.
- إشعارات بريد/واتساب لاحقا.

## أسئلة نقررها لاحقا

- هل الدفع عند الاستلام سيستخدم للكورسات الرقمية فقط، أم لمنتجات/باكدجات معها توصيل؟
- هل نفتح الكورس بعد إنشاء طلب COD أو بعد تأكيد استلام المبلغ؟
- هل المحفظة تقبل شحن رصيد، أم فقط تستخدم للاسترجاعات والكوبونات؟
- هل السعر يكون بالدينار الليبي فقط أم نحتاج عملات أخرى؟
- هل نحتاج منتجات مادية بجانب الكورسات؟
- هل نحتاج فواتير ضريبية أو أرقام طلبات بتنسيق معين؟

## ملاحظات مهمة

- نكمل التتوريال أولا بدون تغيير مسارنا كثيرا.
- أي كود اشتراكات في التتوريال نترجمه إلى منطق شراء كورس أو وصول لكورس.
- لا نبني حلول مالية داخل Sanity.
- Sanity يبقى مصدر المحتوى، وقاعدة البيانات تكون مصدر الحقيقة للطلبات والمدفوعات.
